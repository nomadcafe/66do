# 分析选项（Analytics）检查报告

## 一、当前结构概览

分析页由 **InvestmentAnalytics** 提供，包含 4 个子 tab：

| Tab | 内容 | 数据来源 |
|-----|------|----------|
| **组合 (portfolio)** | 总投资/总收入/净利润、ROI、年化收益、夏普比率；组合表现面积图 | domains + transactions，经时间范围筛选 |
| **表现 (performance)** | 同一张表现图 + 胜率/平均持有期/最大回撤/波动率 | 同上 |
| **风险 (risk)** | 风险等级、分散度、集中度、流动性；投资建议 | 同上 |
| **趋势 (trends)** | 月度收益趋势线、持有/已售域名后缀分布饼图、状态分布饼图、最佳/最差表现域名 | 同上 |

时间范围：3M / 6M / 1Y / ALL，筛选「购买日期」和「交易日期」在该范围内的 domains/transactions。

---

## 二、分析需求是否合理

### 合理且值得保留

- **组合总览**：总投资、总收入、净利润、ROI —— 域名投资核心指标，合理。
- **时间序列图**：投资/收入/组合价值随时间的走势，符合「看趋势」的需求。
- **状态分布**：active / for_sale / sold / expired 占比，便于了解组合健康度。
- **后缀分布**：持有与已售域名的 TLD 分布，对选品和分散有参考价值。
- **最佳/最差表现域名**：基于 ROI 的 best/worst，有参考意义（见下数据问题）。
- **风险维度**：集中度、流动性、简单建议，概念上合理。

### 需澄清或弱化的指标

- **年化收益率 (annualizedReturn)**：公式正确，但当「投资年限」< 1 年时会被放大（例如 2 个月就等价于 (1+r)^6-1），容易误导。建议：投资年限 < 1 年时改为「不显示」或「按当前持有期折算并加说明」。
- **夏普比率 (sharpeRatio)**：当前用「过去 12 个月月度**收入金额**的标准差」当作波动率，再与年化收益率做商，**不是**标准意义上的夏普（应用收益率序列）。数值含义与金融惯例不一致，建议要么改为「月度收益率序列」再算，要么改名为「收益/波动比」并加 tooltip 说明口径。
- **最大回撤 (maxDrawdown)**：当前是对「月度收入金额」序列做回撤，即「收入从高点到低点的相对下降」，**不是**组合净值/累计收益的回撤。建议改名为「月度收入回撤」或改为基于累计收益曲线的回撤。
- **波动率 (volatility)**：同样是「月度收入金额」的标准差（且已乘 100 显示为「%」），量纲与金融里「收益率波动率」不同。建议改名为「月度收入波动」或改为收益率波动并注明单位。
- **流动性风险 (liquidityRisk)**：计算是 `(active+for_sale)/total*100`，数值越高表示可售占比越高、流动性越好，但命名是「风险」，容易理解反。建议改名为「流动性占比」或「可售占比」，并在 UI 上注明「越高越好」。

### 可考虑精简

- **表现 (performance) tab**：与「组合」重复度高（同一张图 + 部分相同指标）。可合并进「组合」或只保留「组合」+「风险」+「趋势」，表现仅作为组合内的一块区域。
- **分散度得分**：当前为 `min(100, domains.length*10)`，仅与域名个数相关，未考虑后缀、金额分布，信息量有限。可保留但加说明「仅按数量」，或后续改为基于 TLD/金额的赫芬达尔等。

## 三、数据与计算需完善之处

### 1. 货币统一

- **问题**：`coreCalculations` 与 `InvestmentAnalytics` 中，所有金额直接相加（`totalInvestment`、`totalRevenue`、`net_amount || amount`），未按 `transaction.currency` 或 `base_amount` 折算到同一货币。
- **影响**：多币种时总投入/总收入/利润/ROI 均失真。
- **建议**：在计算前统一用 `base_amount`（若存在）或按汇率折算到基础货币（如 USD），再参与汇总与图表。

### 2. 时间序列中「投资」与「利润」的口径

- **问题**：  
  - 每月「投资」= 当月月末时点「已持有域名的累计持有成本」（正确）。  
  - 每月「收入」= 当月发生的出售收入（正确）。  
  - 每月「利润」= 当月收入 - 当月累计投资（`revenue - investment`）。这样当月没卖出时利润为负，且与「累计利润」概念不一致，容易误解。
- **建议**：  
  - 要么改为「累计利润」：到当月的累计收入 - 到当月的累计投资；  
  - 要么保留当前为「当月贡献」，但在图例/说明中写清「当月收入 - 当月累计成本」，避免被理解为组合总盈亏。

### 3. 月度收益与波动率/回撤/夏普的口径

- **问题**：  
  - `calculateMonthlyReturns` 返回的是「过去 12 个月每月的**出售收入金额**」，不是「月度收益率」。  
  - `calculateVolatility` / `calculateMaxDrawdown` 基于该金额序列，得到的是「收入的波动/回撤」，不是收益率的。  
  - `calculateSharpeRatio(annualizedReturn, 0.02, volatility)` 中 `volatility` 是金额标准差，与 `annualizedReturn`（百分比）量纲不匹配，夏普数值含义异常。
- **建议**：  
  - 若保留「月度收入」指标：单独命名（如「月度销售收入」），并单独做「月度收入波动/回撤」，不与年化收益率混在一起算夏普。  
  - 若要做标准风险指标：构造「月度收益率」序列（例如每月组合净值或累计收益的环比增长率），再算波动率、最大回撤、夏普比率，并统一在 UI 上注明「基于月度收益率」。

### 4. 年化收益率在短期组合上的表现

- **问题**：`calculateInvestmentYears` 可能 < 1（例如刚买 2 个月），`calculateAnnualizedReturn` 会用 `(1+totalReturn)^(1/years)-1`，导致年化值极大且不稳定。
- **建议**：  
  - 当 `years < 1` 时：不显示年化收益率，或显示为「持有期不足一年，暂不年化」；  
  - 或使用「持有期收益」并注明「若按当前节奏折算为年化约 x%」且仅作参考。

### 5. 域名表现与 best/worst 的口径

- **问题**：`calculateDomainPerformance` 中，对**未售出**域名用 `revenue = domain.sale_price || domain.estimated_value || totalEarned`。即未售出域名按「估值」算收益和 ROI，best/worst 会混入「已实现」和「估值」。
- **影响**：最佳/最差表现中可能包含尚未卖出的域名（基于估值），容易误解为「已实现表现」。
- **建议**：  
  - 方案 A：best/worst 仅基于「已售出」域名（status === 'sold'）的 ROI；未售出不参与排序。  
  - 方案 B：保留当前逻辑，但在 UI 明确区分「已实现 best/worst」与「按估值排序的潜力 best/worst」，或分两块展示。

### 6. 空数据与边界

- **Pie 图**：当所有状态数量为 0（无域名）时，饼图可能为空或报错。建议在「无数据」时隐藏饼图或显示「暂无数据」占位。
- **domainPerformance 为空**：`reduce` 的初始值为 `domainPerformance[0] || { domain: { domain_name: 'N/A' }, roi: 0 }`，在长度为 0 时会使用该默认值，需确认 `domain_name` 在默认对象上存在，避免渲染报错。
- **renewal_count**：`calculateDomainHoldingCost` 使用 `renewal_count`，若为 `undefined` 会变成 `NaN`。建议：`(domain.renewal_count ?? 0)` 或等效默认值。

### 7. 续费与持有成本

- **当前**：持有成本 = 购买成本 + renewal_count × 单次续费成本（仅已发生）。已在 `calculateDomainHoldingCost` 上注明口径，renewal_count 使用 `?? 0` 避免 NaN。  
- **可扩展**：若后续需「预估总持有成本」，可基于 next_renewal_date 增加计划续费；当前逻辑对「已发生」成本是合理的。

### 8. 过期域名损失（Expired Domain Loss）

- **口径**：与持有成本一致，损失金额 = 购买 + renewal_count×续费单价，按**全额冲销**计；未扣减售出回款（若曾部分出售需在交易层单独体现）。
- **无 expiry_date**：`status === 'expired'` 但无到期日时仍计入列表，日期列显示「—」；年度汇总无到期日时用 purchase_date 年，否则归入 `unknown`。

## 四、建议的修改优先级

| 优先级 | 项 | 说明 |
|--------|----|------|
| 高 | 货币统一 | 多币种时用 base_amount 或统一汇率再汇总 |
| 高 | 年化收益 < 1 年 | 不显示或注明「持有期不足一年」 |
| 高 | 夏普/波动/回撤口径 | 要么改为收益率口径并统一命名，要么改名为「收入波动/回撤」并与年化收益解耦 |
| 中 | 流动性命名 | 改名为「可售占比」或「流动性占比」，并注明越高越好 |
| 中 | best/worst 仅已售 | 或明确区分「已实现」与「按估值」 |
| 中 | 时间序列利润说明 | 图例或说明中写清「当月收入 - 当月累计成本」或改为累计利润 |
| 低 | 表现 tab 合并 | 与组合合并或只保留组合+风险+趋势 |
| 低 | 分散度得分 | 加说明「仅按域名数量」或后续改进算法 |
| 低 | 空数据 Pie/0 状态 | 无数据时占位或隐藏图表；renewal_count 默认 0 |

---

## 五、小结

- **需求层面**：组合总览、趋势图、状态与后缀分布、风险维度、最佳/最差表现，对域名投资都有用；部分指标（年化、夏普、波动、回撤、流动性）命名或口径需要与实现一致，避免误导。
- **数据层面**：必须尽快统一货币；时间序列利润、月度「收益」与风险指标的口径需要二选一（要么都按金额，要么都按收益率）并写清说明；年化与 best/worst 的边界情况建议按上表处理。

按上述顺序修复后，分析页在「合理」和「数据完善」上会明显提升。
